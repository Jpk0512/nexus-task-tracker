# Team Routing (persona selection for Nexus)

Persona definitions are configured in `nexus-config.json` and generated into `docs/agents/TEAM.md`. This skill provides the routing DECISION TREE and universal pairing rules that apply to any persona set.

## Cascade routing (model per tier)

Configure model tiers in `nexus-config.json` under `model_cascade`. Defaults:

| Tier | Typical model | Why |
|---|---|---|
| Investigation (Scout) | claude-haiku-4-5 | High-volume read-only exploration; cheap discovery |
| Implementation | claude-sonnet-4-6 | Standard code work; tool-use precision |
| Validation (Lens) | claude-sonnet-4-6 | QA judgment; needs reasoning depth |
| Orchestration (Nexus) | claude-opus-4-7 | Planning / classification / review reasoning |

Project-specific persona models are set in `.claude/agents/<persona>.md` frontmatter after `nexus-install` runs.

## Routing table (generic)

Route by domain ownership defined in `nexus-config.json`. The per-project TEAM.md (generated by `generate-project.py`) is the authoritative routing table after install.

**Universal routing rules (apply to all projects):**

| Work type | Lead | Pair if needed |
|---|---|---|
| Investigation / unknown territory / pre-implementation scouting | Scout | — (read-only, no edits) |
| Validation / acceptance check (after impl) | Lens | — (reports only) |
| Test authoring (stubs before, verification after) | Quill | Coordinates with Lens for coverage |
| Multi-domain or cross-cutting feature | Scout first, then assign by domain | — |
| Work in `[DOMAIN_DIR]` | `[PERSONA]` (from nexus-config.json) | As configured |

**Domain-specific routes** come from `nexus-config.json → personas[].owned_dirs`. After `nexus-install`, consult `docs/agents/TEAM.md` for the full project routing table.

## Pairing rules (universal)

- **New database table** → schema-designer persona designs it; data-layer persona executes the migration
- **New feature spec** → Quill writes failing test stubs BEFORE any implementer begins (Constitution Article I)
- **After every implementer completion** → Lens validates before Nexus marks task done
- **`## NEXUS:REVISE` from Lens** → Re-spawn the original implementer with Lens issues YAML; cap 3 iterations with stall detection
- **Multi-domain work** → Scout first to map the surface, then parallel domain agents

## Forbidden directories (universal pattern)

Every persona's `do_not_touch` list is generated from `nexus-config.json`. Universal constraints:

| Persona | Cannot touch |
|---|---|
| Scout | Anything (read-only: `disallowedTools: Edit, Write, NotebookEdit`) |
| Lens | Anything (`disallowedTools: Edit, Write, NotebookEdit` — reports only) |
| Quill | Non-test files; `.memory/` |
| Nexus | Anything via Edit/Write (`disallowedTools: Write, Edit, NotebookEdit`); orchestrate via delegation only |
| [DOMAIN PERSONA] | All dirs NOT listed in their `nexus-config.json → owned_dirs` entry; `.memory/` |

Consult the generated `docs/agents/TEAM.md` for the full per-persona forbidden-dirs table.

## Classification decision tree

```
Task arrives →
├── Is it a bug fix / config / single obvious change touching ≤2 files (already read)?
│   YES → Simple Task Bypass. Handle inline. No ceremony.
│   NO  → continue
├── Does it span >5 files OR multi-domain OR ambiguous scope?
│   YES → Complex. Spawn Scout first. Then parallel agents per domain.
│   NO  → Standard. Single persona per routing table.
│
└── Standard or Complex →
    1. Run planning gate (skill: nexus-protocol §4)
    2. Reflect (spawn Scout for 5-bullet reflection)
    3. Delegate per CONTRACT.md (skill: contract-schema)
    4. Review completion marker (skill: contract-schema)
    5. Run db_log_cmds
```

## Adding a new persona (after install)

1. Add the persona entry to `nexus-config.json → personas[]`
2. Re-run `python3 scripts/generate-project.py --config nexus-config.json` to regenerate TEAM.md
3. Create the agent file: `.claude/agents/<persona>.md` (use the template from `scripts/templates/persona.md.j2`)
4. Add domain conventions to `.claude/skills/<persona>-conventions/SKILL.md`
5. Update `docs/agents/SKILL_MAP.md` with the persona's required skills
