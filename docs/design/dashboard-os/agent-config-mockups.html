Agent Config — layout options

New Brain nav item under Site Docs. Same disk-mirror idea as Notes / Site Docs, but for global agent homes: ~/.claude, ~/.codex, ~/.cursor, ~/.pi, Oh, plus Claude Desktop (~/Library/Application Support/Claude) and any folder you add. Goal: filter by agent, open the right file fast, edit safely on disk.

Model registered roots → tagged agent → file tree → editor Claude = Code (~/.claude) + Desktop (App Support) Defaults Claude · Codex · Cursor · Pi · Oh Extensible + Add folder with agent tag Safety hide secrets by default · confirm on auth.json
Recommendation: Option B (chip-filtered multi-root library), with Claude registering two roots under one filter. Same Notes shell you already know, filter is first-class, and “Add folder” stays visible. Option D is the best power-user upgrade if you mostly touch the same 6 files every day.

A — Agents rail (Site Docs twin)

Left rail lists Agents instead of Sites. Selecting Claude reveals a compound tree: Code + Desktop. Same Option C shell (list · editor · inspector).

  • Zero new interaction pattern — mirrors Site Docs
  • Claude Desktop is a first-class sibling under Claude
  • Filter is implicit (one agent at a time) — weaker for cross-agent hunting
  • “Add folder” lives one click deeper
Best when consistency with Site Docs matters more than multi-agent scan.

Default roots shown

  • Claude → ~/.claude + Desktop config dir
  • Codex → ~/.codex
  • Cursor → ~/.cursor
  • Pi → ~/.pi
  • Oh → registered path (user-defined)

B — Chip-filtered multi-root library (recommended)

One Notes-style shell. All registered roots live in one library. Agent chips filter the tree. Claude chip shows both Code and Desktop roots. “+ Add folder” always in the header.

  • Filter is explicit — Claude / Codex / All in one click
  • Custom folders are equal citizens (tag them on add)
  • Matches Notes + Site Docs shell
  • Rail can get long with many roots (chips + search mitigate)
Best when you bounce between agents and want add-folder first-class.

Why this wins

Your brief is filter + Claude Desktop + extensibility. Chips solve filter; dual Claude roots solve Desktop; header CTA solves add-folder — without inventing a new IA.

C — Agent homes (cards → dive-in)

Landing grid of agent cards with path health and root count. Click enters tree + editor for that agent. Claude card shows “2 roots · includes Desktop”.

  • Great discovery — clear what lives where
  • Natural place for mount/status badges

  • Cross-agent filter disappears once inside
Best when you want a launcher, less when you live in config all day.

D — Pinned essentials + full tree

Top of the list rail is a curated Essentials strip: the files you actually edit. Full tree below, still filterable by agent.

  • Fastest daily path — skip caches and session junk
  • Pins are cross-agent; filter still narrows them
  • Needs default pin set + “Pin this file”
  • Slightly denser chrome than B alone
Best when 80% of edits are the same six files. Strong upgrade on top of B.

  • Claude · settings.json
  • Claude Desktop · claude_desktop_config.json
  • Codex · config.toml + AGENTS.md
  • Cursor · mcp.json + hooks.json
  • Pi · settings.json

E — Roots registry + workspace

Left pane is a VS Code–style folder registry: path, agent tag, include globs, enabled toggle. Selecting a root opens its tree + editor.

  • Clearest mental model for “add other folders”
  • Easy to disable noisy roots via globs
  • Feels like Settings, not a daily editor
  • Filter-by-agent is a tag, not a one-tap chip
Best when mounting/allowlisting is the hard part. Weaker as everyday edit UI — good companion to B.

Fit with Docker

Registry rows map cleanly to allowlisted host mounts. Useful even if B is the main UI.
